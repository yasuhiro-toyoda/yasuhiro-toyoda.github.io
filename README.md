# Musashi Studio

`musashistudio.com` 向けの GitHub Pages / Jekyll サイトです。

## README に記載すること

- 公開しても問題ない運用ルール
- 記事追加や更新の手順
- タグ管理の方法
- ディレクトリ構成と関連ファイルの場所
- ローカル確認や公開フローの概要

## README に記載しないこと

- ログイン情報や API キーなどの秘密情報
- ドメイン管理サービスや外部サービスの管理画面情報
- 個人情報や内部向けの連絡先
- セキュリティ上の弱点になりうる詳細情報
- 未公開の企画や内部判断メモ

## セットアップ

1. Ruby と Bundler をインストールします。
2. 依存関係を入れます。

```powershell
bundle install
```

3. ローカルサーバーを起動します。

```powershell
bundle exec jekyll serve
```

4. `http://127.0.0.1:4000` を開いて確認します。

## 公開

- `main` ブランチへ push すると GitHub Pages で公開される想定です。
- カスタムドメイン設定は [CNAME](/CNAME) を使っています。

## ブランチ運用ルール

- `main` は公開用ブランチとして扱い、常に公開可能な状態を保ちます。
- 作業は原則として別ブランチで行い、1ブランチ 1 目的にします。
- ブランチ名は `post/...` `fix/...` `chore/...` など、目的が分かる名前にします。
- `main` へ反映する前に、`bundle exec jekyll build` または `bundle exec jekyll serve` で確認します。
- iPad / GitHub ブラウザ編集を前提にした一時運用中は、この README 末尾の「一時運用ルール（iPad・GitHub運用）」を優先します。

## 記事運用ルール

- 公開記事は [`_posts`](/_posts) 配下に Markdown で追加します。
- 公開記事の配置は `_posts/YYYY/MM/YYYY-MM-DD-slug.md` の形式にします。
- 下書きは `_drafts/slug.md` または `_drafts/YYYY/MM/slug.md` に置き、公開時に `_posts/YYYY/MM/YYYY-MM-DD-slug.md` へ移動します。
- 新規記事は [`templates/post.md`](/templates/post.md) をコピーして作成します。
- 新規記事では、ファイル名の日付と front matter の `date` を一致させます。
- `title` `description` `date` `category` `tags` `excerpt` は必ず確認してから公開します。
- 見出しは必要最小限にし、内容がすぐ分かるタイトルを付けます。
- 公開記事として読まれる前提で、個人情報や機密情報は含めません。

## カテゴリ運用ルール

- 利用するカテゴリは [`_data/blog_categories.yml`](/_data/blog_categories.yml) で管理します。
- 記事の `category` には `blog_categories.yml` にある `label` を 1 つだけ指定します。
- 現在の想定カテゴリは `開発` `アプリ情報` `作業効率化` `その他` の 4 つです。
- カテゴリは記事の大区分として使い、細かい話題の整理は `tags` で行います。
- 新しいカテゴリを追加する場合は、既存カテゴリで代用できないか先に確認し、追加後はブログ一覧の絞り込み表示も確認します。

## タグ運用ルール

- 利用するタグは [`_data/blog_tags.yml`](/_data/blog_tags.yml) で管理します。
- 記事の `tags` には `blog_tags.yml` にある `slug` を指定します。
- 新しいタグを追加する場合は、`slug` と表示名の `label` をセットで追加します。
- タグは増やしすぎず、似た意味のものは統合を優先します。

## 更新メモ

- 記事は公開前なら [`_drafts`](/_drafts)、公開後は [`_posts`](/_posts) で管理します。
- 固定ページは `index.html` や [`about/index.html`](/about/index.html) などの HTML ファイルで管理します。
- ツモログの固定ページは [`apps/tsumolog/`](/apps/tsumolog/) 配下にあります。

# 一時運用ルール（iPad・GitHub運用）

## このルールの目的
作業PCの運用を一時的に Gitea 側へ変更している間も、GitHub Pages によるウェブサイト運用を継続するための暫定ルールを定める。  
特に、iPad から GitHub を用いて安全に更新・公開を行うことを目的とする。

---

## 前提
- 公開基盤は **GitHub Pages** を維持する
- 一時的に、作業PCの主な開発・検証環境は **Gitea 側** に寄せる
- iPad は **公開中サイトの保守・更新用端末** として利用する
- iPad での主な操作は、以下を想定する
  - GitHub のウェブサイト上で直接編集する
  - 必要に応じて Codex を補助的に利用する

---

## iPad に期待する役割
iPad では、次の範囲の作業を行う。

### 対応する作業
1. ウェブサイト全体の軽微な修正  
2. ブログ記事の修正  
3. ブログ記事の公開  
4. ウェブサイト全体の小規模な修正  

### 原則として対応しない作業
- 大規模なレイアウト変更
- 複数ファイルにまたがる大幅な設計変更
- 動作確認にローカルビルドや詳細検証が必要な修正
- 依存関係の追加・更新を伴う修正
- 複雑な不具合調査

これらは、原則として **PC 環境復帰後に実施** する。

---

## 運用方針
### 基本方針
- **GitHub ブラウザ編集を主軸** とする
- **Codex は補助** として使う
- iPad では「安全にできる範囲」に作業を限定する
- 無理に広い修正を行わず、事故を防ぐことを優先する

### 役割分担
- **GitHub Pages / GitHub**  
  - 本番公開
  - 軽微修正
  - ブログ修正・公開
- **Gitea / PC**  
  - 本格開発
  - 検証
  - 大規模修正
  - 構成変更

---

## GitHub 上で直接編集してよい作業
以下は iPad + GitHub で対応可とする。

- Markdown の本文修正
- ブログ記事の新規追加
- 誤字脱字修正
- リンク修正
- 画像差し替え
- 軽微な CSS 修正
- 単一ページ内の文言調整
- 設定ファイルの小変更

---

## GitHub 上で直接編集しない作業
以下は iPad から直接対応しない。

- 共通レイアウトの大幅変更
- ナビゲーション全体の再設計
- テンプレート構造の大幅修正
- ビルド結果を見ながら調整が必要な変更
- JS / TS / フレームワーク周りの大きな改修
- サイト全体へ影響する変更で、影響範囲を即時に確認できないもの

---

## Codex の利用ルール
Codex は便利だが、**実行主体ではなく補助役** として扱う。

### Codex に依頼してよい内容
- 修正案のたたき台作成
- 文言の整形
- ブログ記事の下書き補助
- CSS 修正案の作成
- 変更対象ファイルの洗い出し
- PR コメントやコミット文の草案作成

### Codex に丸投げしない内容
- 影響範囲の大きい変更の即時反映
- 構成理解が必要な大規模修正
- 最終確認なしの本番反映
- 「全部まとめて直す」形式の曖昧な依頼

### Codex 利用時の原則
- 提案内容は必ず人間が確認する
- そのまま反映せず、差分と意図を確認する
- 本番反映前に「どのファイルを触るか」を把握する

---

## ブランチ運用
### 原則
- **ごく軽微な修正のみ** 直接反映を許容する
- 少しでも影響範囲が不明な場合は、**別ブランチで作業** する
- 迷う場合は、直接反映しない

### 直接反映してよい例
- 誤字修正
- 記事本文修正
- 画像パス修正
- 軽微なリンク修正

### 別ブランチにすべき例
- CSS の広域修正
- 共通部品の修正
- テンプレートの変更
- ページ横断で影響する修正

---

## ブログ公開ルール
### 記事公開時に確認すること
- タイトル
- 日付
- slug / パス
- カテゴリ / タグ
- 要約
- アイキャッチ画像の有無
- リンク切れの有無
- 公開対象の記事であること

### 公開の基本方針
- 1記事ずつ公開する
- 公開時は他の修正を混ぜない
- 記事追加とサイト構成変更を同時に行わない

---

## コミットルール
### 原則
- コミットは小さく分ける
- 1コミット1目的を意識する
- 後から見て内容が分かる文言にする

### 例
- `Fix typo on top page`
- `Update blog post wording`
- `Add new blog post about xxx`
- `Replace broken image link`
- `Adjust minor CSS for header`

---

## 作業時の判断基準
以下のいずれかに当てはまる場合、その場で無理に修正しない。

- 影響範囲が読めない
- 修正後の表示確認に不安がある
- 複数ファイル修正が必要
- 依存関係やビルド確認が必要
- 元に戻すのが面倒そう
- 「たぶん大丈夫」で進めそうになっている

この場合は、**修正案のメモだけ残して、PC 復帰後に対応する**。

---

## 暫定運用中の優先順位
1. 公開中サイトの安全維持  
2. ブログ修正・公開の継続  
3. 最小限の見た目修正  
4. 大規模改善は後回し  

---

## 結論
一時運用中は、iPad を **公開中サイトの保守・更新専用端末** として扱う。  
GitHub のブラウザ編集を中心に運用し、Codex は補助に留める。  
大きな修正は無理に行わず、PC 環境復帰後に回す。
